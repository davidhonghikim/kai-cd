### 2025-06-20 ‚Äì A2-1 State-Store Flow Analysis

**Stores Reviewed**
1. `serviceStore` (‚âà295 LOC)
2. `viewStateStore` (146 LOC)
3. `settingsStore` (51 LOC)
4. `vaultStore` (105 LOC)
5. `logStore` (39 LOC)
6. `chromeStorage` adapter (14 LOC)

**Hydration Pattern**
‚Ä¢ All stores use `zustand/persist` with a custom `chromeStorage` implementation that proxies to `chrome.storage.local`.
‚Ä¢ Each store sets a `_hasHydrated` flag in `onRehydrateStorage` which components must check.
‚Ä¢ Advantage: prevents render-before-rehydrate race conditions.
‚Ä¢ Gap:  No retry logic if storage read fails (e.g., quota errors).  Recommended to add try/catch + toast.

**serviceStore Logic**
‚Ä¢ Maintains array of `Service` objects + helper for CRUD.
‚Ä¢ URL building centralised in `buildServiceUrl` => good.
‚Ä¢ `getInitialDefaultServices` seeds default services based on localIp/remoteIp from `config`.
‚Ä¢ Adds cloud OpenAI service unconditionally ‚Üí potential duplicate if user adds same URL manually.
‚Ä¢ `checkServiceStatus` (omitted in earlier snippet) uses `apiClient.head`; rate-limited by caller.
‚Ä¢ Side-effect: `updateService` re-computes URL but does **not** update `customUrls` if ipType changes from custom‚Üícloud.

**viewStateStore Logic**
‚Ä¢ Tracks `activeServiceId`, `activeModel`, chat parameters.
‚Ä¢ When `setActiveServiceId` runs, it re-initialises parameters based on the service's `llm_chat` capability.
‚Ä¢ Uses separate `switchToTab` and `switchToPanel` helpers to message background script.
‚Ä¢ Risk: `chatParameters` persisted wholesale; large prompt parameters may bloat `chrome.storage` (5 MB quota).

**settingsStore + vaultStore**
‚Ä¢ `settingsStore` persists theme, log level, dev flags‚Äîsmall, safe.
‚Ä¢ `vaultStore` encrypts credential values with CryptoJS AES‚Äîkey derived from `config.vault.secretKey`; default secret is `default-secret-key` (demo value) if userConfig not set.  Security Risk: Weak default key.

**logStore**
‚Ä¢ Simple in-memory log array, persisted; can grow unbounded ‚Üí quota risk.

**Security Observations**
1. Vault encryption relies on global config secret; need user-specific secret or master password.
2. No schema validation on persisted data; malformed storage could break app.
3. No cross-tab lock when multiple tabs write to same storage key (rare, but async).

**Initial Recommendations** (will become L2 tasks after analysis phase):
‚Ä¢ Add storage size monitoring + auto-purge logs > N entries.
‚Ä¢ Prompt user to set vault master password; derive AES key from it.
‚Ä¢ Wrap storage get/set in try/catch with error toast and fallback defaults.
‚Ä¢ Deduplicate cloud services when user attempts to add an existing URL.
